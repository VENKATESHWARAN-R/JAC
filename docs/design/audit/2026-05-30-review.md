(zero `cli` imports in lower layers, every module documented), real fail-first
discipline, structural `reason:`/HITL enforcement, and a surface-agnostic event

## ⚠️ SESSION-1 POSTMORTEM & HANDOFF (read this first)

**What went wrong with Phase 3 in session 1 (and how to avoid it):**

1. **Root cause — parallel tool-call batching.** I issued many `Edit`/`Bash`
   calls in a single assistant turn. When a linter (ruff via an editor hook)
   rewrote a file mid-batch, every queued `Edit` against the *prior* file
   snapshot silently failed to apply, while the `Bash` test runs in the same
   batch reported failures I then mis-attributed. I committed + pushed **three
   red commits** (`6c53e35`, `7ae37cb`, `5c06c44`, `3c4edaa`) before catching it.
2. **Lesson for next session — go sequential for multi-file changes.** Make ONE
   edit, then run the targeted test/lint in the NEXT turn, read the result,
   then the next edit. Do **not** batch Edits across multiple files in one turn.
   Only batch genuinely independent read-only `Bash` greps.
3. **Always run `uv run pytest tests/ -q` and `uv run ruff check .` and confirm
   GREEN before `git commit`.** Never commit on a partial read of test output.
4. **A real API gotcha discovered (keep this):** `Agent(prepare_tools=...)` is
   **deprecated** in this pydantic-ai version. Use
   `capabilities=[PrepareTools(fn)]` instead (`from pydantic_ai.capabilities
   import PrepareTools`). There is **no** `Agent.prepare_tools` init kwarg and
   **no** `Agent.filtered`; toolsets do have `.filtered(filter_func)`.

## P0 — correctness & security (land before "end stage")

### R1 — Outbound A2A SSRF guard · `DONE` (Phase 1) · 🔴 HIGH
**Landed.** `client._validate_outbound_url` resolves the host (DNS for names)
and refuses private/loopback/link-local/reserved/multicast/unspecified
addresses; called in `a2a_discover` + `a2a_call` only for *unconfigured raw
URLs* (configured named peers are trusted). Knob `a2a.allow_private_peers`
(default `false`) disables it. Tests: 12 in `test_a2a_client.py` (reject
matrix, public-IP allow, knob bypass, hostname→loopback, unresolvable,
raw-URL rejection via both tools, configured-peer exemption via both tools).
Docs: `configuration.md` + `a2a-operator.md`.
**Background.** `a2a_call` / `a2a_discover` issue `httpx` requests to URLs the
*model* supplies, with zero IP validation. A prompt-injected or confused model
can be steered to internal endpoints — cloud metadata (`169.254.169.254`),
`localhost`, RFC-1918. The codebase explicitly cites "SSRF guard" as the reason
it defers *inbound* file URIs (`client.py:694` comment) yet leaves the
model-driven *outbound* path unguarded.
**Evidence.** `capabilities/a2a/client.py:264` (`client.get(discovery_url)`),
`:411` (`client.post(target.url)`). No `ipaddress`/`is_private` checks anywhere.
**Fix.** Add `_validate_outbound_url(url)` resolving the host and rejecting
private/loopback/link-local/metadata ranges (`ipaddress.ip_address(...).is_private
/ is_loopback / is_link_local`), unless a profile knob `a2a.allow_private_peers:
true` is set. Call in both tools before the request.
**Acceptance.** Tests: metadata IP rejected; localhost rejected; public host
allowed; private allowed only with the knob. Configured peers with explicit
private URLs still work via the knob.

### R2 — Resolve `allowed_tools` (dead safety knob) · `DONE` (Phase 3, `f1f1d5c`) · 🔴 HIGH
**Landed (implement, not remove).** `_run_sub_agent` now appends a
`PrepareTools(_make_allowed_tools_filter(packet.allowed_tools))` capability at
the Agent layer when an allowlist is set; the worker sees only the named tools
plus an always-allowed control-plane set `{read_file, ask_main_agent}`. No
allowlist → filter is `None` → unfiltered (common path unchanged). `gru.py`
factory docstring + `summarize-large-files/SKILL.md` updated (no more
over-claim); packet field docstring rewritten. By-construction tests added.
**Phase 1 part landed:** the reusable primitive `tools.restrict_toolset()`
(thin wrapper over pydantic-ai's `.filtered()`) + `FilesystemCapability(allowed=)`
(builds the `FunctionToolset` from only the named functions, so excluded
tools are genuinely unregistered). **Remaining (Phase 3):** wire
`packet.allowed_tools` through `sub_agent_capabilities` so a spawn's toolset
is actually filtered (or remove the field + skill text). The
sub-agent-level acceptance test belongs to Phase 3.
**Background.** The packet field is accepted, threaded through the factory, and
logged to the Logfire span — then **discarded**. Worse, a shipped skill teaches
the model it sandboxes the worker ("narrow the toolset so the sub-agent can't go
exploring"). A sub-agent told `allowed_tools=["read_file"]` still has
`write_file`/`run_shell`. A false safety promise — the worst kind of complexity
(negative value).
**Evidence.** `runtime/sub_agent.py:97` (field), `:483`/`:491` (passed),
`:524` (logged); `runtime/gru.py:150` (`_ = allowed_tools  # reserved`);
`data/skills/summarize-large-files/SKILL.md:38,48` (advertised to model).
**Fix (chosen: implement, not remove — the skill depends on it).** Add a small
`FilteredToolset` (~15 LOC) in `sub_agent_capabilities` that filters the composed
toolset to the named tools when `allowed_tools` is set. Keep `read_file` always
available. If filtering proves harder than expected, fallback is to remove the
field from packet + both factory signatures + the skill text — do **not** ship
the half-wired version.
**Acceptance.** Test: a spawn with `allowed_tools=["read_file"]` has no
`write_file`/`run_shell` in its toolset (assert by construction, not behavior).

### R5 — Extract a `SessionDriver` from `repl.py` · `TODO` · 🟠 HIGH (highest leverage)
**Background.** The entire turn-driving pipeline lives as private functions inside
`cli/repl.py`, fused to `rich`. `_run_turn` constructs a `CliRenderer` *inside

**both** the future browser UI and the SDK layer. The hard part is already done
(surface-agnostic EventBus, Future-based HITL, headless `build_gru` — proven by
the A2A guest path).
**Evidence.** `cli/repl.py:228 _run_turn` (constructs `CliRenderer(console)` at
`:248`), `:342-779 _repl_loop` (437-line function), Gru-assembly block `:387-571`,
budget pre-flight `:809-857`.
**Fix.** Move `_run_turn`, `_recover_failed_history`, `_close_open_tool_calls`,
the budget pre-flight checks, and the Gru-assembly block into a surface-agnostic
`runtime/driver.py::SessionDriver` owning `(gru, bus, session, usage_tracker)` and
exposing `async run_turn(text) -> TurnResult`. The driver only emits onto the bus
and returns — rendering is the caller's job. `repl.py` becomes a thin consumer.
**Sub-items (do as part of, or right after, R5):**

  `EventBus`, `make_approval_handler`, and `SessionDriver` as the documented SDK

- **Note:** EventBus is single-consumer today (`events.py:536`). Multi-surface

## Bidirectional comms — redesign options (R7b working notes)

**Agreed premise.** The feature is worth keeping. The value: a worker that hits a
fork it can't resolve from its packet should *ask the supervisor* (more context)
instead of guessing. Consistent with A2A `input-required` and with the main
agent's `clarify`. Critique is against the *transport*, not the *capability*.

**What's wrong with the current transport (D41 as built).**
- A live sub-agent asyncio task is **parked across main-agent turns**, blocked on
  a queue (`SubAgentChannel`), tracked in a process-global `_pending_channels`
  registry that survives between REPL turns — the heaviest concurrency invariant
  in the codebase.
- Two contextvars + an asyncio race-resolver (`_await_completion_or_question`).
- **Cost can invert the thesis:** when the worker asks, the *main agent re-pays
  its entire (large) context* to compose the answer, and the worker's tokens still
  roll into `session_total`. A chatty 5-round-trip spawn can cost more than doing
  the work inline — defeating the reason sub-agents exist.
- Not persistable/crash-safe (state lives in a live coroutine, not on disk).

**Option A — Suspend/resume state machine (modeled on A2A `input-required`). ★ SELECTED (2026-05-30).**
The sub-agent does not stay alive. When it needs input, the spawn **returns** with
its question *and a checkpoint of its message history*; the spawn enters a
`needs_input` state. The main agent receives the question as the spawn tool's
result, decides how to answer (from its own context, or by calling `clarify` to
ask the human), then **resumes** the sub-agent by re-spawning from the saved
history + the appended answer.
- *Removes:* parked task, global live registry, contextvar dance, race-resolver.
- *Cheaper:* re-pays the **small** sub-agent context on resume, not the large main
  context. (Sub-agents are supposed to be small — that's the whole point of
  isolation — so this is strictly the cheaper re-pay.)
- *Crash-safe / persistable:* a `needs_input` spawn is a record on disk, resumable
  across `--resume`. Aligns with the session model.
- *Expandability bonus:* a pending question is a structured **state**, not a live
  coroutine — a browser UI or SDK can render and answer it without being attached
  to the live process. Composes cleanly with R5's `SessionDriver`.
- *Unifies the mental model:* "a sub-agent question = the sub-agent task entering
  input-required" — identical to A2A. One concept, two surfaces.
- *Cost:* still bounded (keep a round-trip cap); the human-escalation path is free
  because the main agent just calls the existing `clarify`.

**Option B — Single-shot `escalate_to_supervisor`, no conversation.**
Narrow the surface to one question → one answer → continue (like `clarify`), no
multi-turn channel, no `/spawns` listing. Simpler than today, but doesn't cover
the "needs a short back-and-forth" case the user described. Could layer on top of
A as the *tool shape* while A is the *transport*.

**Option C — Richer pre-flight packet (complementary, not a replacement).**
`success_criteria`/`forbidden_actions`/`expected_output` already exist to make
spawns self-sufficient. Strengthen prompt guidance so questions only arise for
genuinely unforeseeable forks. Reduces *frequency*, doesn't remove the need.

**Option D — Plan-then-confirm.** Worker returns a plan for approval before
executing. Handles "wrong path," not "missing info discovered mid-execution."
Orthogonal; possible future add.

**Proposed direction (for discussion):** adopt **A** as the transport, expose it
as a small **B-shaped tool** (`ask_supervisor(reason, question, context)`) so the
worker's surface stays a single clear verb, let the main agent answer-or-`clarify`
(escalate to human) at its discretion, keep a round-trip cap, and lean on **C**
(prompt guidance) to keep it rare. This preserves 100% of the user-facing value,
removes the parked-task/global-registry/race machinery, fixes the cost-inversion,
and makes pending questions persistable + GUI-renderable.

**Open questions to settle before R7b:**
1. Does pydantic-ai expose a clean "resume an Agent run from saved history +
   appended message" path, or do we re-`run` with reconstructed history? (Spike.)
2. Round-trip cap: keep 5, or tie it to `max_turns`?
3. Human-escalation default: always let Gru answer first, or a packet flag
   (`escalation: supervisor | human`) the spawner sets?
4. Rendering: reuse the existing `SubAgentQuestion`/`SubAgentAnswer` events (good)
   but drive them from state transitions, not a live channel.
