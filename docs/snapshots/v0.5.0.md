```bash
# Persistent (recommended)
uv tool install "git+https://github.com/VENKATESHWARAN-R/JAC.git@v0.5.0"
jac --help

# One-off
uv run --from "git+https://github.com/VENKATESHWARAN-R/JAC.git@v0.5.0" jac --help
```

## 5. CLI surface

**What it is.** A Typer command + an interactive REPL with slash commands. Every slash line is handled locally and never sent to the model.

**Setup.** None beyond `jac init`.

**Sample prompts.**

```bash
jac --help
jac --profile NAME              # one-shot profile
jac --model anthropic:claude-opus-4-6   # raw model override
jac --resume                     # latest session in this project
jac --session 2026-05-24T14-30-00
jac sessions                     # list project sessions
```

In-REPL slash:

```text
/help            ← lists every slash command
/sessions
/resume          ← latest if no id
/clear           ← new session
/profile         ← list
/profile claude  ← switch
/model           ← numbered picker from active tier list
/model openai:gpt-4o    ← explicit override
/budget          ← table of limits vs usage
/tokens          ← per-counter breakdown
/skill list
/spawns          ← parked bidirectional sub-agents
/a2a status
/exit
```

**What success looks like.** Status bar updates as profile/model change. `/profile NAME` with a missing key fails loudly (`JacConfigError`) and *rolls back* — Gru remains on the previous profile, not in a half-broken state.

**Common breakages.**
- Silent default model (should never happen — see CLAUDE.md "Fail-first").
- `/clear` losing the prior session (it should remain on disk).
- Slash command being interpreted by the model (means routing broke).

## 6. Profiles & tiers

**What it is.** Each profile = `{tiers: {small, medium, large}, active_tier, env, requires_env}`. The main agent always asks for a **tier** (never a specific model). Profiles are stored in `~/.jac/config.yaml`.

**Setup.** `jac init` creates one. To add a second:

```bash
jac init                # wizard adds another profile
# or hand-edit:
jac profiles edit claude
```

Minimal YAML:

```yaml
default_profile: claude
profiles:
  claude:
    active_tier: medium
    tiers:
      small:  [anthropic:claude-haiku-4-5]
      medium: [anthropic:claude-sonnet-4-5]
      large:  [anthropic:claude-opus-4-6]
    env: {}
```

**Sample prompts.**

```bash
jac profiles
jac profiles list
jac profiles use claude
jac profiles edit claude        # opens $EDITOR; validate-on-save
jac profiles remove claude      # stored keys kept
```

In REPL:

```text
/profile                ← list
/profile ollama-local   ← switch (rebuilds Gru; rolls back on missing keys)
/model                  ← pick concrete model from active tier list
```

**What success looks like.** `jac keys` shows `✓` per required env var for each profile. `/profile NAME` switching prints the new active tier + model in the next status-bar render.

**Common breakages.**
- Tier cascade going *down* (e.g. `medium` requested but only `small` configured → must error, not silently downgrade).
- Profile switch leaving credentials from the previous profile leaked in the env.

## 7. Sessions & resume

**What it is.** One conversation thread per `<repo>/.agents/sessions/<timestamp>/`. After every completed turn, `messages.json` is rewritten; `plan.json` persists alongside.

**Setup.** None — sessions auto-create per project.

**Sample prompts.**

```bash
jac sessions
jac --resume
jac --session 2026-05-24T14-30-00
```

In REPL:

```text
/sessions
/resume                  ← latest
/resume 2026-05-24T14-30-00
/clear                   ← new session (old kept on disk)
```

**What success looks like.** `--resume` prints `(resumed, N prior messages)` in the greeting. If a `plan.json` existed with `in_progress` steps, they reload as `pending` (the actor was interrupted).

**Common breakages.**
- Mid-turn crash erasing prior turns (must not — only completed turns are committed).
- `--resume` with no sessions silently launching a new one (should be a clear error).

## 8. Memory (the 2×2 matrix)

**What it is.** Two scopes × two ownerships = four files loaded into Gru's instructions every run.

|  | **User scope** (all projects) | **Project scope** (this repo) |
| --- | --- | --- |
| **User-authored** (JAC never writes) | `~/.jac/AGENTS.md` | `<repo>/AGENTS.md` |
| **JAC-managed** (via `remember`/`forget`, HITL) | `~/.jac/memory.md` | `<repo>/.agents/memory.md` |

**Setup.** None — files bootstrap on first `remember` call. You can pre-author `AGENTS.md` yourself.

**Sample prompts.**

```text
» Remember as a user preference: "always use uv run, never bare python".
» Remember as a project convention: "this repo prefers ripgrep over grep for codebase searches".
» Forget the user preference about uv run.
```

**What success looks like.** Approval panel renders with `scope`, `category`, `content`. After approval, the relevant `memory.md` gains a new bullet under the right section with an HTML audit comment (`<!-- jac: <iso-ts> session: <id> -->`). Duplicate `remember` calls in the same section are rejected (exact-normalized match).

**Common breakages.**
- `remember` succeeding without approval (HITL bypass).
- `remember(scope=project)` succeeding outside a git repo (must fail).
- `forget` silently matching multiple lines (must error and ask for narrower content).

## 9. Tools

**What it is.** Every tool requires `reason: str` as its first argument (enforced via `@jac_tool` decorator). The reason is shown in the approval panel.

**Setup.** None.

**Sample prompts.** See §16 benchmark L1–L4 for graduated exercises. One per family:

```text
» Read pyproject.toml lines 1-30.                        ← read_file
» List the contents of src/jac/.                          ← list_dir
» Grep for "spawn_sub_agent" in src/.                    ← grep
» Find all *.md files under docs/.                        ← glob
» Run `just check` and tell me if it passes.              ← run_shell (approval)
» Start a background process: uv run python -m http.server 8765. ← start_process (approval)
» Search the web for "pydantic-ai sub-agent pattern".    ← web_search
» Fetch https://ai.pydantic.dev/ and summarize it.       ← fetch_url
» Plan: [draft architecture sketch, run tests, write summary]. ← plan
» I'm unsure which approach to take — clarify by asking me: A) refactor, B) rewrite, C) leave alone. ← clarify
```

**What success looks like.** Writes / shell / processes show approval panels. Reads / searches / web do not. Background processes show `[task tail-N]` panels when output flows. `plan` re-renders the checklist on every update.

**Common breakages.**
- Tool missing `reason:` (should be impossible — fail at agent construction).
- `run_shell` truncating large output instead of routing through the summarizer (was a v0.2 bug; fixed in v0.3.0).

## 10. Human-in-the-loop (HITL)

**What it is.** A unified approval panel for every potentially destructive tool. Includes `write_file`, `edit_file`, `run_shell`, `remember`, `forget`, `start_process`, `kill_process`. The panel title now shows **who's asking**: `approval needed · Gru` or `approval needed · minion-N`.

**Setup.** None.

**Sample prompts.** Any prompt that triggers a write. To force one:

```text
» Edit README.md and add the word "TEST" to the very end. Don't apply if I deny.
```

**What success looks like.** Panel shows the tool name, `reason`, the diff (for `edit_file`) or the full args. Answer with `y`, `n`, or a denial reason (the reason is fed back into the agent's history so it can adapt).

**Common breakages.**
- Approval bypass on any of the gated tools.
- Denial reason not surfacing in the next agent turn (means the deny-with-feedback path broke).

## 11. Cost controls

**What it is.** Five mechanisms that keep Gru's context — and your bill — small.

1. **Tool-result post-processor** — opted-in tools (`run_shell`, `web_search`, `fetch_url`) above `cost.tool_result_threshold_tokens` (8000 default) are routed through the profile's **small-tier** model and replaced with a summary. Original saved to `.agents/cache/tool-results/<session>/<call-id>.txt`. Only fires when the small tier is *strictly cheaper per output token* than the current tier (per `providers.yaml`).
2. **Prompt-cache-friendly assembly** — system prompt header is byte-stable across turns; only memory.md + per-turn user prompt invalidate the cache.
3. **Compaction** — token-budget-aware auto-summarization at 70% of `compaction.max_context_tokens` (200k default).
4. **Token budgets** — opt-in `session_input` / `session_total` / `project_total` with warn (80%) and hard-stop (100%) thresholds.
5. **Sub-agents** — see §12. Heaviest lever.

**Setup.** Defaults are sensible. To exercise on a low budget, drop in `<repo>/.agents/config.yaml`:

```yaml
budget:
  session_total_tokens: 50000
  warn_pct: 50
  hardstop_pct: 90
compaction:
  max_context_tokens: 100000
  refuse_pct: 80
cost:
  tool_result_threshold_tokens: 4000   # easier to trip
```

**Sample prompts.**

```text
» Fetch https://docs.pydantic.dev/latest/concepts/types/ and summarize the type categories.
» Run `uv run python -m pytest tests/ -x -q` and report the first failure if any.
/tokens
/budget
/budget extend 50000          ← raise session_total limit
```

**What success looks like.** `/tokens` shows three optional lines when activity warrants them:

```
summarize:   calls=N  original=…  summary=…  saved=…  (small-tier spent in=… out=…)
cache:       read=…  write=…  hit-rate=…%
sub_agents:  small=…  medium=…  large=…
```

A summarized tool result starts with `[AI-summarized via <model> — original NNN tokens, full output at <path>]`. The path is real — `read_file` on it returns the unsummarized bytes.

**Common breakages.**
- Post-processor firing on `read_file` (must not — line numbers / exact text would be destroyed).
- Cache hit-rate ≈ 0% on a long session with Anthropic (means prompt prefix got contaminated again, regression of the day-granularity datetime fix).

## 12. Sub-agents (the headline of v0.5.0)

**What it is.** A single `spawn_sub_agent(reason, task_summary, tier, task_packet)` tool plus a parallel fan-out `spawn_sub_agents(reason, task_summary, spawns)`. Each worker is a fresh agent with its own message history; only the final string returns to the main loop.

**Setup.** None — works with any profile that has more than one tier.

### 12.1 Sequential `spawn_sub_agent`

```text
» Use spawn_sub_agent to read the three reference skills under src/jac/data/skills/ and tell me which is largest.
```

**What success looks like.** Approval panel shows resolved tier (with cascade note if any), `task_packet` shape, tool allowlist. After approval: model runs in isolation, returns one string + token counts. `/tokens` `sub_agents:` line gains a row.

### 12.2 Parallel `spawn_sub_agents`

```text
» For each Phase A/B/D/E feature, find the docs page and write a one-paragraph summary. Use spawn_sub_agents with one worker per phase.
» Compare how src/jac/tools/filesystem.py, src/jac/tools/search.py, and src/jac/tools/shell.py each register their tools. Spawn one minion per file; aggregate the findings.
```

**What success looks like.**
- Approval renders a **per-spawn summary table** (index / label / tier / one-line objective) — NOT a JSON dump.
- As each worker starts, a blue panel `▶ minion-N` appears.
- As each finishes, a green line `✓ minion-N done · turns=N`.
- Final aggregated answer comes back as one tool result; `/tokens sub_agents:` line includes all workers.
- Depth cap = 1 holds: sub-agents do **not** see `spawn_sub_agent` or `spawn_sub_agents` in their toolset.

### 12.3 Bidirectional ask/respond (D41, on by default)

```text
» Spawn a minion to draft a new section for docs/user-guide/examples.md about A2A. It should ASK ME if it's unsure about section title or tone before finalizing.
```

**What success looks like.**
- Yellow panel "question from minion-N" appears mid-run; spawn pauses.
- Status bar gains `spawns:1`; `/spawns` lists the parked spawn with its question.
- Your reply renders back as a cyan panel; the minion continues.
- Hard cap = **5 round-trips**. The 6th `ask_main_agent` returns a directive: "finalize with what you have, list any open uncertainties as discrepancies." It does NOT error.

**Opt-out:**

```yaml
cost:
  sub_agent_bidirectional: false   # default is true
```

**Common breakages.**
- Tier cascade going down (small→nothing should error; medium→small must not happen).
- Sub-agent succeeding without approval.
- Per-spawn lifecycle events arriving all-at-once at the end (means parallelism broke — they should stream as workers start/finish).
- Bidirectional 6th ask raising an exception instead of returning the finalize directive.

## 13. Skills

**What it is.** Markdown playbooks with YAML frontmatter that Gru reads on demand. Skills are *advice, not execution* — they give instructions for how to use existing tools.

**Setup.** None — three reference skills ship with the package. To add your own:

```bash
mkdir -p .agents/skills/repo-tour
cat > .agents/skills/repo-tour/SKILL.md <<'EOF'
---
name: repo-tour
description: 5-minute tour of this repo for new contributors.
---
# Repo tour
1. Read README.md and CLAUDE.md.
2. List src/jac/ and identify top-level subpackages.
3. Read docs/architecture.md §5 (locked decisions).
4. Output a labeled tour as a numbered list.
EOF
```

**Sample prompts.**

```text
/skill list                   ← active + shadowed tables
» Review my last commit using the code-review skill.
» Summarize src/jac/cli/app.py — use the summarize-large-files skill if it fits.
/skill use verify-change      ← inject as next user turn
/skill reload                 ← rescan all three dirs; reports +/- counts
» Give me the repo tour.      ← exercises the custom skill above
```

**What success looks like.**
- `/skill list` shows 3 reference skills (`code-review`, `summarize-large-files`, `verify-change`) plus any you added.
- Gru's system prompt lists skill names + descriptions (up to 2 KB; name-only fallback if longer).
- When relevant, Gru calls `load_skill(name="…")` rather than improvising.

**Common breakages.**
- Same-named skills both loading (project should shadow user should shadow package; shadowed entries should still appear in `/skill list`).
- 2 KB system-prompt skills block being unbounded (regression — would tank cache hit rate).
- `/skill reload` not reflecting a freshly-saved SKILL.md (means file watching is broken).

## 14. A2A (Phase 4, v1 scope)

**What it is.** Inbound *guest Gru* (read-only) plus outbound `a2a_discover` / `a2a_call` tools. Bidirectional file transfer via `FileWithBytes`.

**Setup.** Server is opt-in — `jac` does not auto-start it.

### 14.1 Loopback exercise

Terminal A:

```text
/a2a serve
/a2a token              ← copy
```

Terminal B (separate `jac` session, may be the same repo):

```text
/a2a peer add self http://127.0.0.1:8001 --bearer
  bearer token: <paste>
» Use a2a_call on peer self to ask: "What's the entrypoint of this codebase?"
» Use a2a_call on peer self with files=["./README.md"] and ask: "Summarize this file."
```

### 14.2 Headless server

```bash
jac a2a serve                                # bearer auth (default)
jac a2a serve --port 9000
jac a2a serve --unsafe                       # no auth — trusted networks only
```

### 14.3 Demo peer (data-analyst)

```bash
cd examples/data-analyst-a2a
export ANTHROPIC_API_KEY=sk-...
export ANALYST_BEARER=$(openssl rand -hex 24)
uv run server.py
```

```text
# in another terminal's JAC
» /a2a peer add analyst http://127.0.0.1:8002 --bearer
  bearer token: <paste $ANALYST_BEARER>
» Use a2a_call on peer analyst with files=["./examples/data-analyst-a2a/sample-data.csv"]
  and ask "What's the revenue trend across the year? Plot it."
```

**What success looks like.**
- `[a2a out →]` / `[a2a out ✓]` panels on the caller side.
- `[a2a in ←]` / `[a2a in ✓]` panels on the server side.
- Returned task's `_jac_saved_files` lists paths under `.agents/a2a/inbound-files/<task_id>/`.
- `/a2a status` shows last 5 inbound calls + tokens_used.
- `inbound.jsonl` gains a new line per inbound call.

**Common breakages.**
- Outbound peer returns `submitted` state and `a2a_call` returns immediately (must poll `tasks/get` until terminal).
- Inbound file ending up outside `<repo>/.agents/a2a/guest-uploads/<context_id>/` (path traversal regression).
- Filename collisions silently overwriting (should get numeric suffixes).
- `--unsafe` working without an explicit flag (no auth must require explicit opt-in).

## 15. Observability

**What it is.** Logfire spans on every model call, tool call, sub-agent spawn, memory write, and A2A call. Spans carry `template`, `task_id`, `parent_run_id`, `token_cost`, `duration`, `exit_status`.

**Setup.** Set `LOGFIRE_TOKEN` to ship spans to the cloud. Without it, tracing is local (still visible in `logfire-spans.log` if configured).

**Sample prompt.** Any benchmark run. Then in the Logfire UI, filter to spans with `pydantic_ai.run` and inspect the tree — sub-agent runs are children with the parent chain visible.

**What success looks like.** Every sub-agent has a `parent_run_id` pointing back to the main run. Summarize spans (`tool_summarize`) show `original_tokens`, `summary_tokens`, `saved_tokens`, `cache_path`. A2A calls appear under fasta2a span family.

**Common breakages.**
- Sub-agent spans missing `parent_run_id` (parent chain broken).
- `tool_summarize` spans missing on a session where `/tokens` shows summarize activity.

## 16. Benchmark prompts (graduated complexity)

Standardized prompt set. **Save each transcript** (`/sessions` for the id, then copy `messages.json`). For each benchmark capture: **pass y/n**, **wall-clock time**, **`/tokens` snapshot**, and **anything surprising**.

### L1 — Trivial (single tool, no reasoning)

```text
» Read the first 30 lines of pyproject.toml and tell me the project name and Python version.
» List the top-level directories under src/ (exclude hidden).
» Grep for "TODO" in src/ and report counts per file.
```

Watch: tools called exactly once; no plan or sub-agent overhead; reads need no approval.

### L2 — Simple (multi-tool, short reasoning)

```text
» Give me a 3-bullet overview of what this repo does. After answering, remember "this repo is X" as a project fact.
» Find the file that defines the main CLI entrypoint, read it, and summarize the registered subcommands.
» Save my preference: "always use uv run, never bare python" — scope=user.
```

Watch: `remember` approval panel renders; memory.md gets a new bullet with audit comment.

### L3 — Medium (cross-file synthesis + plan)

```text
» Map every public tool Gru exposes to the file where it's defined. Use the plan tool to track progress and produce a table at the end.
» This codebase has a "capability" pattern. Pick three capabilities, read them, and explain the contract they share.
» I want to add a new slash command called /hello that prints a greeting. Walk me through what I'd need to change, but don't make edits yet.
```

Watch: `plan` checklist renders and updates in-place; no premature edits.

### L4 — Complex (edit + verify + HITL)

```text
» Add a docstring to the main() function in src/jac/cli/app.py describing what it does. Run `just check` after.
» Fix any ruff warnings in src/jac/tools/. Run `just fix` first; only apply manual edits for things ruff can't auto-fix.
» Add a single pytest test that asserts `jac --version` exits 0. Place it in tests/ following the existing naming convention.
```

Watch: `edit_file` approval shows the diff hunks; `run_shell` for `just check` may trip the post-processor (visible in `/tokens summarize:`).

### L5 — Parallel sub-agents (`spawn_sub_agents`)

```text
» For each Phase A/B/D/E user-facing feature, find the docs page and write a one-paragraph summary. Use spawn_sub_agents with one worker per phase.
» Read the three reference skills under src/jac/data/skills/ in parallel sub-agents and tell me which body is largest and why.
» Compare how src/jac/tools/filesystem.py, src/jac/tools/search.py, and src/jac/tools/shell.py each register their tools. Spawn one minion per file; aggregate.
```

Watch: per-spawn summary table in the approval panel (not JSON); `▶ minion-N` start panels stream as workers begin; `✓ minion-N done · turns=N` on finish; `/tokens sub_agents:` populates.

### L6 — Bidirectional sub-agent (ask/respond)


## 19. Known limitations in v0.5.0

- **No MCP loader** — Phase F is next; today, external tools mean writing a JAC tool or a sub-agent.
- **No Plan Mode** — Phase G; `plan` tool exists but there's no read-only planning mode.
- **A2A streaming** — fasta2a 0.6.1 doesn't support `message/stream`; card declares `streaming: false`.
- **A2A `FileWithUri`** — outbound URI fetching deferred until SSRF story exists; inbound URI parts silently skipped.
- **No YOLO mode** — every destructive tool is HITL-gated. Acceptable trade-off until v2 sandbox lands.
- **Windows untested** — macOS / Linux only.
- **No auto memory extraction** — user-tier memory.md is hand-edited or written via `remember` only; predict-calibrate extraction is deferred to v2.
